# MMIS 查詢流程與 Debug 指南

這份文件整理 MMIS 查詢型自動化在實作與除錯時最容易踩到的規則。重點不是單一 skill 的腳本細節，而是未來所有 MMIS Playwright / 批次查詢都可重用的操作知識。

## 1. 查詢流程與操作規則

### 1.1 先切換 `所有記錄`，再輸入查詢條件

- 在 MMIS list app 中，若不先切換 `所有記錄`，查詢結果可能受目前頁面狀態或既有篩選影響。
- 正確順序是：
  1. 進入 app
  2. 點查詢下拉按鈕
  3. 選 `所有記錄`
  4. 等待畫面與 filter row 穩定
  5. 再開始填欄位

實作原則：

- 不要把 `所有記錄` 視為只需切一次的全域狀態。
- 在每次查詢前重新確認目前畫面已處於 `所有記錄` 狀態，比盲信既有狀態更穩。

### 1.2 Enter 觸發查詢前，欄位必須先取得 focus

- 在 MMIS filter row 中，`Enter` 不是只要送出去就會生效。
- 若欄位沒有 focus，`Enter` 可能不會觸發資料過濾。

穩定做法：

1. 填完所有查詢欄位
2. 再點一下最後要觸發查詢的欄位
3. 確認該欄位已成為 active element
4. 再送 `Enter`

建議：

- 優先用 `locator.press("Enter")`
- 不要只用 `page.keyboard.press("Enter")` 而不確認 focus

### 1.3 查詢條件輸入順序要固定

對 MMIS list filter row 來說，穩定順序應固定，不要每次隨意變更。

建議順序：

1. 先清空會殘留預設值的欄位
2. 再填固定條件欄
3. 再填日期欄
4. 最後填車號或主查詢欄
5. 最後在主查詢欄按 Enter

這樣做的原因：

- 可減少欄位殘值干擾
- 比較容易在 debug log 中核對
- 比較不會出現查詢條件寫錯位置卻不自知

## 2. 欄位輸入與資料格式規則

### 2.1 日期要先正規化，再輸入查詢欄

Excel 中的日期常是：

- `2026/4/5`
- `datetime`
- 或其他非最終查詢格式

查詢前必須先轉成 MMIS 實際可用格式。

#### 規則

- 標準格式：`YYYY/MM/DD`
- 需要比較條件時，再加前綴，例如 `>YYYY/MM/DD`

例子：

- Excel：`2026/4/5`
- 查詢：`>2026/04/05`

不要接受：

- `>2026/4/5`
- `>>2026/04/05`

### 2.2 Excel 與系統格式不一定相同

不要把 Excel 看到的字串直接當成可輸入 MMIS 的最終值。

開發時必須先做：

- trim
- 日期格式統一
- 必要運算子補齊
- 最後再送進 filter row

### 2.3 有預設值的欄位要用鍵盤清除

對某些 MMIS 欄位，單純 `fill("")` 不一定能讓系統狀態真的回到空值。

較穩定的清除方式：

1. click 欄位
2. `Ctrl+A`
3. `Backspace`
4. 必要時再 `Delete`
5. 驗證欄位值是否真的變成空字串

若欄位仍有值，不要繼續查詢，先當成輸入失敗處理。

## 3. 網頁元素定位策略

### 3.1 優先使用可驗證的欄位 mapping

在 Maximo filter row 中，`C:n` 常比 class 更有意義。

例如：

- `C:1`
- `C:2`
- `C:3`
- `C:11`

若已知欄位對應意義，應明確記錄並重用。

### 3.2 不要只依賴 class

- Maximo class 常重複出現在多個欄位
- class 較不適合辨識特定欄位語義

優先順序：

1. 已驗證的欄位 id 片段
2. 結構位置
3. 穩定文字
4. class 作為最後輔助

### 3.3 判斷資料列是否存在，要區分三種情況

1. 查詢有結果
2. 查詢無結果
3. 畫面仍停留在舊結果，尚未真正更新

第三種情況最容易誤判。

因此查詢後不能只看：

- `result row count > 0`

還要同時確認：

- loading overlay 已消失
- 無資料文案是否出現
- 第一列內容是否真的更新

## 4. 常見錯誤與解法

### 4.1 查詢無結果，但其實是條件填錯位置

問題現象：

- 畫面看起來有填值
- 查詢永遠回不到預期資料

常見原因：

- 車號填進錯誤欄位
- 預設值欄位未清空
- 日期欄位或比較條件填錯位置

解法：

- 在 Enter 前把關鍵欄位實際值記錄到 log
- 必要時對照截圖確認欄位位置

### 4.2 Enter 無效

問題現象：

- 條件明明已填
- 但畫面結果沒有真正刷新

原因：

- 送 Enter 前，查詢欄位沒有 focus

解法：

- 點回最後觸發查詢的欄位
- 驗證 active element
- 再送 `Enter`

### 4.3 沒切換 `所有記錄`

問題現象：

- 查詢命中率不穩
- 不同列的結果像是被上一個查詢影響

原因：

- 查詢範圍仍受既有 list 狀態限制

解法：

- 每次查詢前明確切 `所有記錄`
- 切換後等 filter row 再次可見

### 4.4 查詢後誤判為「狀態未穩定」

問題現象：

- 程式等待很久
- 最後判定 timeout
- 但實際上畫面已顯示「沒有要顯示的列。」

原因：

- 無資料文案判斷不完整
- 只抓 `查無資料` 或 `沒有資料`，漏掉 Maximo 真正使用的文案

解法：

- 把實際畫面文案納入無結果判斷
- 例如：`沒有要顯示的列。`

### 4.5 查詢後誤用上一筆殘留結果

問題現象：

- 這一筆明明應該無資料
- 卻誤點了上一筆查詢留下的第一列

原因：

- 只看「有沒有結果列」
- 沒檢查該結果列是否為新的查詢結果

解法：

- 記錄查詢前第一列 signature
- 查詢後再比較第一列是否真的改變
- 同時搭配無資料文案判斷

## 5. Debug 與驗證策略

### 5.1 查詢前截圖

若要 debug 查詢條件填入問題，最有價值的截圖時機是：

- 所有欄位填完之後
- 按 Enter 之前

這能直接確認：

- 日期是否已加 `>`
- 車號是否填在正確欄位
- 有沒有預設欄位殘值

### 5.2 關鍵欄位值一定要寫 log

至少要記：

- `C1`
- `C2`
- `C3`
- `C11`

目的：

- 不用只靠肉眼看截圖
- 可以快速知道欄位值是否真寫進 DOM

### 5.3 單筆測試要先通，再放大到批次

建議流程：

1. 先挑一筆已知應命中的資料
2. 驗證單筆條件
3. 確認能穩定查到指定工單
4. 再套回整批流程

典型用途：

- 驗證日期格式
- 驗證欄位 mapping
- 驗證 Enter 是否真的觸發查詢

### 5.4 批次流程要容忍單筆失敗

批次自動化的目標不是每筆都完美，而是：

- 單筆失敗不拖垮整批
- 失敗原因可回溯
- 可以續跑未完成列

建議：

- 每列即時寫回 Excel
- 失敗列寫明確狀態
- 續跑時只補跑空白列或明確失敗列

## 6. 已驗證的 1A 批次勾稽規則

以下是已驗證可用的 1A 查詢規則：

- `C1 = 新竹機務段`
- `C2 = 空字串`
- `C3 = 車組/車號`
- `C11 = >YYYY/MM/DD`

已驗證命中案例：

- `新竹機務段 + 890 + >2026/04/05 -> 115-1A-26331`
- `新竹機務段 + 709 + >2026/04/07 -> 115-1A-25616`

已驗證的無資料情況：

- 畫面會出現 `沒有要顯示的列。`
- 這是有效業務結果，不是程式錯誤

## 7. 開發建議總結

對 MMIS 查詢型 automation 來說，最重要的不是把流程跑起來，而是：

- 查詢條件要填在正確欄位
- 查詢前後狀態要可驗證
- 無資料與未更新要分開判斷
- 單筆驗證要先打通
- 批次時要可續跑

當這五件事都有做到時，MMIS 批次查詢的穩定性才會真的上來。
